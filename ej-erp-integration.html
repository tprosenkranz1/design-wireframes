Service architecture, data flow, and UI filtering. Updated after 4/1 meeting with Mike Hack.

1. Service architecture

EJ / EMQUE side Access channel Cru backend Out of MVP

How to read this: one linear pipeline. Pull from EMQUE, stage raw, parse the description, normalize, match, store a snapshot. The UI reads from snapshots. Plaid feeds the match step separately on the bank side. All six steps are new build work - Match and Normalize reuse logic from the current CSV flow, but wiring them to the EMQUE source is fresh.

2. Data flow

Automated step (overnight) Stored result What the user sees

The pipeline in one sentence. Pull 5 columns, stage them, parse vendor and invoice out of the description, shape into the CSV format, match against bank lines, store the snapshot.

3. UI filtering mechanism

Active / selected Secondary control Sync action

Answer to "can someone just hit a date picker and be done with it?" Yes. One primary control: a date range button that opens a preset list + custom calendar. Day granularity, no timestamps. Sync status is a small piece of text elsewhere in the header, not part of the filter. Account filter stays as a secondary dropdown for the case where someone wants to isolate one account.

Target CSV columns (from general_ledger_template.csv): Date, Description, Debit, Credit, Journal, Vendor Name, Client Name, Job, Inv #. Nine fields. Nothing else.

Summary: 9 fields. 5 are a straight pass-through from EMQUE. 2 come from parsing the description field. 2 are open questions for Mike (Client Name and Job).

Description parser pattern (from template sample): {VENDOR} [CK: {check#}] [INV: {invoice#}]. Vendor is the leading segment before "CK:" or "INV:". Check # lives inside the description, not as a separate CSV column - so we only need to extract it for the Vendor parse, not as its own output field.
