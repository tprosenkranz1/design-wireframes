Cru Bookkeeping: ERP + Bank Automation (rev. 2026-04-23, post backend-model verification)

Both sides of the reconciliation: GL from EMQUE and bank transactions from Plaid. Service architecture, data flow, UI filtering, and field mappings. Bank pipeline grounded against real Django models: PlaidToken, PlaidAccount, PlaidTransaction.

1. GL side service architecture

EJ / EMQUE side Access channel Cru backend Out of MVP

How to read this: one linear pipeline. Pull from EMQUE, stage raw, parse the description, normalize, match, store a snapshot. The UI reads from reconciliation state. Bank lines come in via a parallel pipeline from Plaid (see section 2). All six steps are new build work - Match and Normalize reuse logic from the current CSV flow, but wiring them to the EMQUE source is fresh.

2. Bank side service architecture

EJ bank side Plaid Already in Cru Cru backend

How to read this: PlaidTransactionsSyncService (shipped as CRU-276) already talks to Plaid's /transactions/sync endpoint using a cursor stored on PlaidToken.transactions_cursor, writes rows into the PlaidTransaction table, and fires off webhook-driven deltas. Our new work is a reader that pulls from that table into the Bookkeeping Agent, normalizes to the 4-column bank CSV, classifies against the 18 rules, and hands rows to the same matching engine the GL side uses. Both pipelines write into the shared reconciliation state the UI reads from.

No parse step needed here (unlike the GL side). The classification engine runs text-based rules against the raw narrative (PlaidTransaction.name), and the matcher uses narrative + amount + date. That's enough to classify and match without splitting vendor/invoice out of the text. If we later need a clean check number for reference data, we read PlaidTransaction.payment_meta (JSON) and fall back to a narrative regex - a check extractor, not a full parser.

Current state caveat: today the Bookkeeping Agent reads bank data from manual xlsx uploads, not from the Plaid tables. The target state lifts the xlsx step out of the critical path and keeps it as a fallback for banks Plaid doesn't cover and for historical backfill beyond Plaid's retention window.

Things the reader has to filter: pending=true rows (exclude until posted), is_from_constrafor=true rows (Constrafor-originated payments, should not hit client rec), and the company's skip logic on update_transactions (EJ will need should_update_plaid_data_daily=True so syncs actually run daily, not monthly).

3. Data flow (both pipelines converging at the matcher)

GL lane (EMQUE) Bank lane (Plaid) Shared matcher User view

The flow in one sentence. Two lanes, same matcher. GL pulls from EMQUE, parses narratives into vendor/invoice, normalizes. Bank reads PlaidTransaction rows that PlaidTransactionsSyncService has already pulled from Plaid, normalizes, classifies against 18 rules. Both lanes hand rows to the same matching engine (CRU-307), which updates the shared reconciliation state the UI reads from.

Why the bank lane is one step shorter. No parse step. Classification works on the raw name narrative, the matcher works on narrative + amount + date. We don't need to dig vendor or invoice numbers out of the narrative to make classify/match work. Check number, if needed downstream for reference lookups, lives in payment_meta JSON - we'd read it there, not parse the narrative.

What's already in prod. Classify (CRU-306, CRU-498), match (CRU-307), and the current CSV flow into both. What's new is wiring the two Pull+Stage+Normalize lanes in front of them.

5. Bank CSV fields: what we need vs. what the PlaidTransaction model gives us

Target CSV columns (from bank_statement_template.csv): Narrative, Post Date, Credit Amount, Debit Amount. Four fields visible to the user, plus four internal fields attached during ingest (Classifications, General Ledger Tag, Ledger Tag Status, Account). Sources below reference real Django model columns verified against payments/models/plaid_transaction.py.

CSV column PlaidTransaction source Status What we do with it
Available directly from PlaidTransaction rows (4 of 4 visible columns)
Narrative PlaidTransaction.name Direct CharField(512). Pass through raw - this is the field the classification rules run against. merchant_name isn't stored as a column on our model; if we ever want it, pull from payment_meta JSON.
Post Date PlaidTransaction.date Direct DateField (posted date). Format MM/DD/YYYY. Do NOT use authorized_date - that's the card auth, not the settle. Both columns exist on the model.
Credit Amount PlaidTransaction.amount (inflow) Sign-split Single DecimalField(11,2) column in our DB. Plaid uses positive for outflows by default. If the amount represents an inflow for this account, it goes in Credit; otherwise Debit. One number becomes two columns in our format.
Debit Amount PlaidTransaction.amount (outflow) Sign-split Same column-split logic as Credit, just the other side.
Internal fields attached during ingest (4 more, not shown in CSV)
Classifications rules engine output Computed 18 deterministic rules run per-row against Narrative and Account. Unmatched rows get tagged "unmatched". Already in production (CRU-306, CRU-498).
General Ledger Tag matcher output Computed Link to the matched GL row (or set of rows in 1:many). From the existing matching engine (CRU-307).
Ledger Tag Status matcher output Computed matched / unmatched / partial. Drives the unmatched-items queue in the UI.
Account PlaidAccount FK (PlaidTransaction.account_id) Mapped Resolve the PlaidAccount row's account_name to our display name (Operating, Disbursement, etc.) via the per-company account mapping. PlaidAccount FK is nullable on the model - need to handle that case.
Other PlaidTransaction fields we care about
Pending filter PlaidTransaction.pending Filter BooleanField on the model. Today's CSV flow has no concept of pending. Default plan: exclude pending=true until posted. Open question for the team.
Constrafor-originated filter PlaidTransaction.is_from_constrafor Filter BooleanField. Flags transactions that originated from inside Constrafor (not client bank activity). Almost certainly need to exclude these from the reconciliation view. Confirm with Evgeniy.
Check # (if needed) PlaidTransaction.payment_meta (JSON) TBD There is NO discrete check_number column on our PlaidTransaction model - only a payment_meta JSONField that may contain Plaid's check_number if the bank supplied it. For MVP we don't need this (classification + matching don't use check#). If a future use case needs it, read payment_meta["check_number"] first and regex the narrative as fallback.

Summary: 4 visible columns populated from PlaidTransaction (2 pass-through, 2 sign-split). 4 internal columns computed by the existing classification + matching engines. Two filters we need during ingest: pending=true (exclude) and is_from_constrafor=true (exclude).

Correction versus earlier draft: prior version claimed Plaid gives us check_number as a discrete field. Verified against payments/models/plaid_transaction.py - it doesn't. No such column on our model. If check number matters downstream, it's a payment_meta JSON lookup with narrative regex fallback. For MVP it doesn't matter: classification and matching work off narrative + amount + date.

Current state caveat: PlaidTransactionsSyncService (CRU-276, shipped Nov 2025) already populates the PlaidTransaction table via cursor-based sync and webhooks. The Bookkeeping Agent for EJ still reads xlsx uploads today. Net-new work is the Bookkeeping Agent reader + filters, not the Plaid sync itself.

6. UI filtering mechanism

Active / selected Secondary control Sync action

Answer to "can someone just hit a date picker and be done with it?" Yes. One primary control: a date range button that opens a preset list + custom calendar. Day granularity, no timestamps. Sync status is a small piece of text elsewhere in the header, not part of the filter. Account filter stays as a secondary dropdown for the case where someone wants to isolate one account.
